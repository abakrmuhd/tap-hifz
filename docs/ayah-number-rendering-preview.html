Ayah Number Rendering Decision

Real `mushaf-layout` data shows that the Arabic verse number is appended to the final word token of each ayah. Tap Hifz should split that token visually: the word remains selectable text, and the verse number becomes the tappable ayah control.

Rendering Rule

Use the dataset as the source of truth, but separate the verse number from the final word when rendering.

1. Render by page line. Iterate lines[] and preserve line boundaries.

2. Render text lines by word. For each text line, render words[] instead of the pre-joined text.

3. Detect verse-ending tokens. If a word ends with an Arabic verse number, split it into the Quran word text and an ayah number control.

4. One ayah control per ayah. Long ayahs spanning many lines get only one tappable ayah number, at the actual verse end.

5. No duplicate controls. If a visible page contains a partial ayah but not its verse-ending number, that page does not invent a second ayah number.

Decision: the tappable control is the rendered ayah number itself, derived from the final word token. It maps to surah:ayah, while the surrounding word remains selectable text.

Interaction Consequence

This resolves the main ambiguity without changing the mushaf text layout.

Tap ayah number: increment ayahProgress[surah:ayah].

Long press ayah number: open ayah count details and correction controls.

Long press word: keep normal/selectable text behavior.

Double tap ayah number: increment page-local transition only if the previous ayah number is also visible on the same page.

For page 48, Ayah 2:282 has one tap target at the end of the page. The app should not add repeated 282 controls to every line.
